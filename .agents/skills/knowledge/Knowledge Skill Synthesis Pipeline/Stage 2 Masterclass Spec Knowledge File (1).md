# Stage-2 Meta Master Class – Conceptual Merging Guide  
_For GPT B: Stage-2 XML Synthesizer (Knowledge Document)_

> NOTE:  
> This document is **conceptual guidance only**.  
> It explains *how to think* about merging multiple masterclasses into one coherent model.  
> The actual **output format and XML schema** are defined in the system Instructions of GPT B and must always override any implied structure here.

---

## 1. Purpose of Stage-2 (Conceptual)

Stage-2’s job is to:

- Take **2–5 curated masterclass summaries on the same topic**.
- Understand each one’s:
  - Core thesis / worldview,
  - Mechanisms and frameworks,
  - Heuristics and anti-patterns,
  - Implementation patterns and style.
- **Merge** them into:
  - One unified conceptual model for the topic,
  - With a **small set of canonical concepts and rules** that can drive an agent’s behavior.

Think of Stage-2 as building the **“brain spec”** for a future skill:
- What it believes,
- How it thinks,
- How it decides,
- What it avoids,
- How it sounds.

The XML `<patch>` defined in the Instructions is just the **container** for that brain.

---

## 2. Input Assumptions

Each Stage-1 masterclass summary has been:

- Generated by a Stage-1 Masterclass Synthesizer.
- Reviewed and refined by a human (you).
- Structured into sections such as:
  - Big picture / thesis,
  - Modules,
  - Frameworks / processes,
  - Principles & heuristics,
  - Mistakes / anti-patterns,
  - Implementation ideas,
  - Style / teaching flavor.

All masterclasses in a Stage-2 run:

- Share the same **Meta Topic** (e.g. “Email launch campaigns”, “Sleep optimization fundamentals”).
- May **overlap**, **disagree**, or emphasize different aspects.
- Should be treated as different “lenses on the same territory”.

---

## 3. Mental Model for Merging

When merging multiple masterclasses, follow this mental model:

1. **Core Thesis Layer**  
   - What do they collectively say is *true* about this topic?
   - What do they agree on?
   - Where do they maintain different but compatible emphases?

2. **Mechanism Layer**  
   - What are the key **mechanisms / lenses** explaining how things work?
   - Which mechanisms are essentially the same under different names?
   - Which are truly distinct?

3. **Framework Layer**  
   - What are the main **step-by-step frameworks / processes**?
   - Which ones solve similar problems?
   - Which ones should be merged vs kept separate?

4. **Heuristic Layer**  
   - What rules-of-thumb guide decisions?
   - Where do they align or conflict?

5. **Anti-Pattern Layer**  
   - What do they warn you NOT to do?
   - Which mistakes appear in more than one source?

6. **Behavior Layer (for Agents)**  
   - If this were a person, how would they:
     - Start a conversation on this topic?
     - Diagnose a situation?
     - Choose which framework to apply?
     - Avoid common traps?
     - Explain things in a human way?

7. **Human Flavor Layer**  
   - What metaphors, story patterns, and values define the *feel* of this topic?

All these layers get compressed into the **XML fields** described in the system Instructions.

## 4. Practical Steps for Merging

### Step 1 – Extract Core Theses

From each masterclass:

- Identify its primary thesis and big promises.
- Identify recurring beliefs like:
  - “This works because…”
  - “Most people fail because…”

Then:

- Distill a **shared thesis**: points that are obviously agreed upon.
- Add **distinctive thesis points** only if they meaningfully change the way a skill would:
  - Diagnose,
  - Decide,
  - Act,
  - Explain.

**Goal:** 2–5 unified thesis points that reflect the *combined* view.

---

### Step 2 – Identify and Canonicalize Mechanisms

Mechanisms are “how this really works” stories:
- e.g. “Deep Sleep Deficiency Loop”, “Campaign Ecosystem Thinking”, “Micro-Commitment Ladder”.

From all masterclasses:

1. List all mechanisms / key lenses.
2. Group similar ones:
   - Same idea, different names.
3. Choose **canonical names** that are:
   - Short,
   - Clear,
   - Descriptive.

For each canonical mechanism, note:

- What it explains.
- Why it matters for decisions.
- When it is most relevant (what signals suggest its use).

These become `<mechanism>` entries in XML.

---

### Step 3 – Merge Frameworks

Frameworks are step-by-step structures and processes.

Process:

1. Extract all explicit frameworks:
   - Named models,
   - Processes,
   - Checklists.
2. Spot near-duplicates:
   - Similar steps with slightly different wording or order.
3. Decide for each group:
   - **Merge** into one improved framework, OR  
   - Keep separate if they truly solve different problems.

When merging:

- Retain:
  - The clearest steps,
  - Any valuable diagnostic checks,
  - Any decision branches that matter.
- Simplify:
  - Remove redundant steps,
  - Combine micro-steps into clean phases.

Keep only **3–7 canonical frameworks** that:
- A future agent would actually use regularly.
- Cover most use-cases under the Meta Topic.

These map to `<framework>` elements in XML with:
- `name`,
- `summary`,
- `when_to_use`.

---

### Step 4 – Consolidate Heuristics

Heuristics are compact decision rules.

From all masterclasses:

- Extract statements like:
  - “If you have X, don’t start with Y.”
  - “Always clarify A before choosing B.”
  - “When in doubt, do C rather than D.”

Normalize them into:

- `IF [condition / signal / context] THEN [action / framework / priority].`

Then:

- Remove redundancy.
- Resolve conflicts:
  - If two rules clash:
    - Convert them into a **conditional pair**:
      - Use rule A when condition 1 is true, rule B when condition 2 is true.
- Keep **5–10** heuristics that:
  - Actually change decisions,
  - Cover common situations,
  - Are clear enough to implement as logic.

These become `<heuristic>` entries in XML.

---

### Step 5 – Distill Anti-Patterns

Anti-patterns are recurring mistakes or self-sabotage moves.

From all masterclasses:

- Gather myths, warnings, “don’t do this” lists.
- Group similar ones.
- Focus on **3–7 high-impact anti-patterns** that:
  - Are common,
  - Are costly,
  - Or cause confusion for agents.

For each:

- Give it a short label (“Shiny Object Hopping”, “Overloading Beginners”).
- Describe:
  - The behavior,
  - Why it is harmful,
  - What to do instead.

These map to `<anti_pattern>` elements in XML.

---

### Step 6 – Agent Behavior Rules

Agent behavior rules are the operational SOPs for the future skill.

From all masterclasses:

- Look at Implementation / “How to apply” sections.
- Look at recommended:
  - First questions to ask.
  - Sequences of actions.
  - Ways to respond in tricky situations.
- Think:
  - “If this were a smart assistant, how should it *always* approach this topic?”

Write **5–10 concise rules** that answer questions like:

- How to **start**:
  - What to clarify first.
- How to **choose frameworks**:
  - Which ones are defaults vs. advanced.
- How to **handle uncertainty**:
  - When to present options,
  - When to encourage experimentation.
- How to **avoid harm**:
  - Where caution is needed.

These become `<rule>` entries under `<agent_behavior_rules>` in XML.

---

### Step 7 – Humanizing Elements

This layer keeps the skill from feeling generic.

From the masterclasses:

- Extract:
  - Metaphors (“treat this like a lab, not a courtroom”),
  - Story arcs (e.g. “struggle → insight → experiment → breakthrough”),
  - Values (e.g. patience, curiosity, self-compassion).

Pick **1–5** that:
- Capture the tone and attitude you want future agents to embody.
- Are easy to paraphrase in explanations.

These map to `<humanizing_elements>` in XML.

---

## 5. General Priorities

When synthesizing:

- **Less but better.**
  - Each field in XML should encode something **decision-changing**, not trivia.
- Favor items that:
  - Clarify what to do,
  - Clarify what NOT to do,
  - Clarify *how* to think about the topic.
- “Nice-to-have” and low-leverage details belong in the original masterclass docs, **not** in the patch.

The Stage-2 XML Synthesizer’s **system Instructions** specify the exact XML format and limits (counts, lengths). This document is here to support:

- Good judgment,
- Deep integration,
- Consistent conceptual design.

The Instructions are the authority on **what to output**.  
This document is guidance on **how to think while choosing what to encode**.
