## EXECUTIVE SUMMARY

The Framework Extractor is the **abstraction engine** of the ULTRAMIND self-improving loop. It takes raw deconstruction reports (from sales pages, funnels, emails, ads) and converts specific winning examples into reusable pattern frameworks that can be injected into skills as upgrade patches.

**Core Value:** Turn competitors' best work into your system's permanent intelligence.

## OPERATING MODES

### Mode 1: Single Pattern Extraction
```yaml
purpose: Extract one specific pattern from a deconstruction
input: DECONSTRUCTION_REPORT + pattern_focus
output: Single FRAMEWORK_CARD

use_when:
  - You spotted something specific ("that hook was genius")
  - Quick extraction during review
  - Building pattern library incrementally

process:
  1. Identify the specific technique
  2. Abstract from example to skeleton
  3. Define context (when/where it works)
  4. Format as FRAMEWORK_CARD
```

### Mode 2: Full Deconstruction Mining
```yaml
purpose: Extract ALL learnable patterns from a deconstruction
input: Complete DECONSTRUCTION_REPORT
output: FRAMEWORK_LIBRARY (5-15 patterns)

use_when:
  - Analyzing a major winner
  - Building comprehensive pattern sets
  - Deep competitive intelligence

process:
  1. Scan entire deconstruction
  2. Tag every extractable technique
  3. Prioritize by impact/reusability
  4. Abstract each to framework
  5. Cross-reference with existing patterns
  6. Output complete library
```

### Mode 3: Cross-Winner Pattern Synthesis
```yaml
purpose: Find "super patterns" that appear across multiple winners
input: Multiple DECONSTRUCTION_REPORTs (3-10)
output: SUPER_PATTERN_REPORT + high-confidence frameworks

use_when:
  - Quarterly pattern review
  - Identifying industry-wide best practices
  - Building "canonical" techniques

process:
  1. Load multiple deconstructions
  2. Pattern-match across all
  3. Score by frequency (appears in X of Y winners)
  4. Extract super patterns (3+ appearances)
  5. Flag as high-confidence frameworks
```

### Mode 4: Skill Gap Analysis
```yaml
purpose: Compare extracted patterns against existing skill capabilities
input: FRAMEWORK_LIBRARY + TARGET_SKILL
output: GAP_REPORT + PATCH_RECOMMENDATIONS

use_when:
  - Before generating upgrade patches
  - Skill audit/improvement cycles
  - Prioritizing what to add

process:
  1. Load target skill's current frameworks
  2. Compare against extracted patterns
  3. Identify gaps (patterns skill doesn't have)
  4. Identify upgrades (better versions of existing)
  5. Prioritize by impact
  6. Recommend patches
```

## EXTRACTION METHODOLOGY

### The Abstraction Ladder

```
LEVEL 1: SPECIFIC EXAMPLE
"I wasn't going to share this strategy. I've kept it private 
for 11 years. But after what happened last Tuesday in 
Washington, I can't stay silent anymore..."

         ↓ ABSTRACT ↓

LEVEL 2: STRUCTURAL SKELETON
[Reluctance statement] + [Time held private] + 
[External trigger event] + [Moral obligation to share]

         ↓ ABSTRACT ↓

LEVEL 3: NAMED FRAMEWORK
"The Reluctant Expert Open"
- Establish reluctance to share
- External event forces hand
- Magnitude hinted, not revealed
- Urgency through exclusivity

         ↓ ABSTRACT ↓

LEVEL 4: UNIVERSAL PRINCIPLE
"Perceived sacrifice increases perceived value"
"External triggers create urgency without manipulation"
```

## OUTPUT FORMATS

### FRAMEWORK_CARD
```yaml
framework_id: "FW-2026-001"
name: "The Reluctant Expert Open"
category: "hooks"
subcategory: "authority_openers"

origin:
  source: "Agora Financial - Palm Beach Letter"
  extracted_from: "sales_page_deconstruction"
  extraction_date: "2026-01-15"
  performance_indicators: "$50M+ campaign"

pattern:
  structure: |
    1. [Reluctance Statement] - "I wasn't going to..."
    2. [Time Held Private] - Duration of secrecy
    3. [External Trigger] - Event forcing revelation
    4. [Moral Obligation] - Why sharing now
    5. [Magnitude Hint] - Tease without reveal
  
  skeleton: |
    "I [reluctance]. I've [time private]. 
    But [external trigger], [obligation to share]..."
  
  variables:
    - reluctance_reason: Why they held back
    - time_period: How long kept secret
    - trigger_event: What changed
    - stakes: Why it matters now

psychology:
  why_it_works:
    - Scarcity (was private)
    - Authority (held valuable info)
    - Urgency (external trigger)
    - Trust (reluctance = not salesy)
  
  neurochemical: "Dopamine (curiosity) + Oxytocin (trust)"

application:
  when_to_use:
    - Market sophistication: Stage 4-5
    - Positioning: Insider/expert
    - Offer type: Information, financial, high-ticket
    - Traffic: Cold to warm
  
  when_not_to_use:
    - Stage 1-2 markets (too sophisticated)
    - Mass market brands (feels elitist)
    - Transparency-first positioning
    - When you lack genuine credentials
  
  variations:
    - "The Accidental Discovery" (stumbled upon vs. developed)
    - "The Whistleblower" (obligation to expose)
    - "The Deathbed Reveal" (legacy motivation)

examples:
  original: |
    "I wasn't going to share this strategy. I've kept it 
    private for 11 years..."
  
  adapted_health: |
    "I never planned to go public with this protocol. 
    For 8 years, it stayed in my private practice..."
  
  adapted_business: |
    "This framework was supposed to stay internal. 
    We've used it for 47 clients without ever revealing it..."

integration:
  target_skills:
    - advertorial_copy_master
    - sales_page_copywriter
    - vsl_script_writer
  
  layer: L2
  section: "Hook Formulas > Authority Hooks"
```

## FAILURE MODES

### FM1: Over-Abstraction
```yaml
symptom: Pattern so abstract it's not actionable
example: "Use good words" instead of specific structure
root_cause: Went too high on abstraction ladder
recovery: Drop back to skeleton level, keep structure
prevention: Always include concrete example + skeleton
```

### FM2: Under-Abstraction
```yaml
symptom: "Pattern" is just the original example rephrased
example: Copying exact words instead of extracting structure
root_cause: Didn't climb abstraction ladder
recovery: Ask "what's the skeleton here?"
prevention: Require structure + variables format
```

### FM3: Context Blindness
```yaml
symptom: Pattern applied where it doesn't fit
example: Reluctant Expert used for Stage 1 mass market
root_cause: Didn't capture when_to_use / when_not_to_use
recovery: Add application context
prevention: Require context fields in every FRAMEWORK_CARD
```

### FM4: Duplicate Patterns
```yaml
symptom: Same pattern extracted multiple times with different names
example: "Reluctant Expert" vs "The Hesitant Authority"
root_cause: Not checking existing library
recovery: Merge and consolidate
prevention: Cross-reference before adding new patterns
```
